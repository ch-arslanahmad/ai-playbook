# Context Files

Every AI coding tool reads a project-level markdown file at startup.

They all do roughly the same thing,

- set rules, explain project structure, context and related agents, skills, commands but each tool has a different filename.

---

## AGENTS.md, the pseudo-standard

`AGENTS.md` is read by **OpenCode**, **Gemini CLI**, **Cursor**, **Codex**, and **Claude Code**.

```markdown
---
agents:
  - name: reviewer
    description: Code review specialist
    prompt: Focus on security and edge cases

commands:
  - name: tdd
    description: Run test-first workflow
    prompt: Write tests before implementation
---

Project rules here...
```

`AGENTS.md` does not have a one defined structure, but it is recommended to have a YAML frontmatter section for structured config and a markdown body for free-form instructions, however it is mostly generated by the tools themselves, so knowing that is enough.

## Per-tool context files

All the tools who natively support `AGENTS.md` are:

- OpenCode
- Charm's Crush
- Cursor
- Codex
- Claude Code
- Antigravity CLI (Gemini)

## Exceptions

- `.github/copilot-instructions.md` (locally) or `AGENTS.md` by Copilot,
- Project-level rules live in `.cursor/rules/*.mdc` or `AGENTS.md` at the repo root.

## Global Files

| Tools           | Global?                                             |
| --------------- | --------------------------------------------------- |
| Claude Code     | `~/.claude/CLAUDE.md`                               |
| Antigravity CLI | `~/.gemini/config/AGENTS.md`                        |
| OpenCode        | `~/.config/opencode/AGENTS.md`                      |
| Charm's Crush   | `~/.config/crush/AGENTS.md`                         |
| Codex           | `~/.codex/AGENTS.md`                                |
| Copilot         | No global file documented                           |
| Cursor          | **Settings → Rules** (docs don't specify file path) |

**Cursor is the odd one out** — User Rules are set through **Settings → Rules** in the UI. The official docs don't mention a global config file path.

## Symlink strategy

`AGENTS.md` covers most tools natively. Claude Code is the odd one out (supports in modern versions however, CLAUDE.md is better), Claude naturally prefers reading `CLAUDE.md`. So, a solution may be:

```bash
ln -s AGENTS.md CLAUDE.md
```

One file, both tools happy. Keep frontmatter lean if Claude Code reads it (it ignores YAML but the markdown body works fine). Or put Claude-specific instructions behind a `## Claude Code` section header.
